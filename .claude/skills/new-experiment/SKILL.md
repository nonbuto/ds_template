# /new-experiment スキル

## このスキルの役割

実験を「目的のある問い」として始めるためのスキル。
手順を走らせる前に「この実験で何を明らかにするか」「何が起きたら方針を変えるか」を言語化する。
言語化した内容は実験記録として残り、`/kaggle-submit` での振り返りとつながる。

---

## 実行手順

### フェーズ1: 直近の学びを確認する（思考層）

`experiments/log.csv` の直近エントリを読み込む。

- 直近の実験名・OOF・LBスコア・learning（学び）を表示する
- learning が未記入の場合は「前の実験の振り返りが未完了です。`/kaggle-submit` で学びを記録してから新実験を始めることを推奨します」と案内する（強制はしない）
- OOFとLBの差分（乖離）が大きい場合は「OOF改善がLBに反映されていない実験が続いています。今回の実験はその原因仮説を検証するものですか？」と問いかける

### フェーズ2: 今回の実験の目的を明確化する（思考層）

以下の3問をユーザーに問いかける。
**3問すべてに短くてもよいので答えてもらってから次へ進む。**

---

**Q1: この実験で何を明らかにしたいですか？**
（例1 — 初回ベースライン: 「数値カラムのみでCV/LB相関は成立するか。以降の改善判断の基準点を作る」）
（例2 — 作業用HP調整: 「FE計測ノイズを減らすために、軽いOptuna（20〜30試行）で作業用HPを決める」）
（例3 — 段階的FE: 「〈特徴量X〉を1列追加するとOOFがΔいくら改善するか」）
（例4 — 本格HP最適化: 「確定した特徴量セットでOptuna 100試行以上を行い、性能上限を探る」）
（例5 — モデル多様性: 「〈モデルY〉はLGB/CBと予測パターンが異なるか（OOF相関<0.998か）」）
→ 「XをするとYが改善するか」という形で表現できると理想的。手段ではなく目的を書く。

> HP最適化実験の場合は現在のステージを確認する:
> Stage 3（作業用）→ 目的はノイズ低減、試行数は少なくてよい
> Stage 5（本格）→ 特徴量セット確定後のみ実施、試行数は多く

**Q2: どんな結果なら「この方向で正しい」と判断できますか？**
（例: 「OOFが+0.0002以上」「LBが現在ベストを超える」）
→ 事前に基準を決めておくことで、結果の解釈が恣意的になることを防ぐ。

**Q3: どんな結果なら「中止・別の方向へ」と判断しますか？**
（例: 「OOFが悪化する」「LBが-0.00005以上下がる」）
→ 撤退基準を事前に決めておく。

---

ユーザーの回答を受けて、AIは以下を確認する:
- Q1が「何をするか（手段）」になっていないか → 「何を明らかにするか（目的）」に言い換えを促す
- Q2/Q3の基準が曖昧な場合は具体的な数値に落とすよう促す

### フェーズ3: インフラ整備（実行層）

1. `git status` でworking treeがcleanか確認（汚れている場合は「コミットしてから進めてください」と警告して停止）

2. **ブランチ作成（条件付き）**

   Q1の回答から実験の性質を判断し、以下の基準でブランチを作るかどうかを決定する:

   | `exp/` ブランチを作る | `comp/<competition>` のまま進む |
   |---|---|
   | 新アルゴリズムの追加（XGB/NN/RF など） | FEの1列追加・仮説検証 |
   | 特徴量セットの大幅再設計（列数±20%以上） | HPチューニング（Optuna） |
   | CV戦略の変更 | ブレンド重みの調整 |
   | Stackingなどアーキテクチャ変更の試験 | バグ修正・スクリプト整備 |

   → 上記左列に当てはまる場合のみブランチを作成する:
   ```bash
   git checkout -b exp/<実験名>
   ```
   → 当てはまらない場合は「この実験は現在のブランチで進めます（ブランチ作成不要）」と明示して続行する

3. **実験番号の重複チェック**（衝突防止）
   ```bash
   ls experiments/runs/ | grep "^exp" | sort | tail -5
   tail -3 experiments/log.csv | cut -d',' -f2
   ```
   未コミットの `experiments/runs/exp{NNN}_*` が存在する場合、その番号は使用済みとして扱い、より大きい番号を使う。

4. `src/config.py` の `EXPERIMENT_NAME` を更新

5. `experiments/log.csv` に今回の実験行を予約追記する
   - `experiment_name`, `experiment_question`（Q1の回答）, `success_criteria`（Q2）, `abort_criteria`（Q3）を記録
   - スコア列は空欄のまま（`/kaggle-submit` で埋める）

6. **`SESSION.md` を更新する**（セッション引き継ぎのため）
   - `現在のステージ`: 現在の Stage 番号と状況
   - `最後に完了したこと`: 今回の実験開始を記録（スコアは後で埋める）
   - `次にやること`: 「<実験名> の学習を実行する」を先頭に記載
   - `現在の主要スコア`: 最新のOOF tuned / LB / OOF-LB乖離 / 実験IDを1テーブルで上書き更新

### フェーズ4: スタートへの接続

以下を表示して完了:

```
実験「<実験名>」を開始します。

目的: <Q1の回答>
成功基準: <Q2の回答>
中止条件: <Q3の回答>

【コミットのタイミング】
  学習完了・OOFスコア確認直後にコミットしてください。
  形式: feat(expNNN): <実験の目的>
        OOF=<score>  model=<model>  features=<feature_set>
  ※ tracker.end_run() がコミットメッセージ案を自動出力します

SESSION.md を更新しました。次のセッションで /resume を実行すると
この実験の目的と次のアクションが復元されます。

実験完了後は `/kaggle-submit` で提出・振り返りを行い、
学びを記録してから次の実験を始めてください。
```

---

## 設計の意図

- 実験の「目的」は結果が出てから決めると合理化が起きる。事前に記録することが重要。
- 撤退基準を先に決めることで、悪い結果が出たときに「もう少し試せば」と追加実験を重ねる罠を防ぐ。
- Q1〜Q3への回答はそのまま `experiments/log.csv` の narrative として残り、後から「なぜあの時そう判断したか」を追跡できる。
