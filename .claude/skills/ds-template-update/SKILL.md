# /ds-template-update スキル

## このスキルの役割

コンペ作業中に発見した改善点をTODO_TEMPLATE.mdに記録する。
**単なるタスク追加ではなく、「テンプレートとして何を学んだか」を問う場**。

コンペ固有の知見がそのままテンプレートに流れ込まないよう、
「この改善はなぜ価値があるか」「他のどんな文脈でも使えるか」を確認する。

---

## 実行手順

### フェーズ1: 汎用性の峻別（思考層）

ユーザーが提案した改善を受けて、以下を問いかける。

**Q1: この改善が必要だと気づいた「具体的な場面」を教えてください**

（例: 「特徴量を追加したときに、過去に試した組み合わせかどうか分からなくなった」）

**Q2: この改善は、次の3分類のどれに当てはまりますか？**

| 分類 | 説明 | 例 |
|---|---|---|
| **A: 汎用プロセス** | どんなDS問題でも繰り返す思考・手順 | 「仮説を立ててからFEする」「実験目的を事前記録する」 |
| **B: 技術インフラ** | 実装は具体的だが抽象化すれば汎用 | 「log.csvにlearning列を追加」→ 抽象: 「実験の学びを記録する仕組み」 |
| **C: コンペ固有** | このデータ・この評価指標でしか意味がない | 「num_leaves < 150のガイドライン」「CB38列の特徴量リスト」 |

- **A → テンプレートの核心として記録・実装する**
- **B → 抽象化した形でテンプレートに記録する**（具体的な実装はコンペブランチに残す）
- **C → コンペのドキュメント（COMPETITION.md）に記録。テンプレートには含めない**

---

### フェーズ2: 記録の作成（実行層）

分類に応じて記録先を変える:

**A・Bの場合 → `TODO_TEMPLATE.md` に追記**

```markdown
## [YYYY-MM-DD] <優先度> — <改善タイトル>
- **説明**: <何を改善するか>
- **発見のきっかけ**: <どんな場面で必要だと気づいたか>
- **汎用性の根拠**: <なぜどんなDS問題でも使えるか>
- **抽象化の方向**: <具体的な実装をどう一般化するか>
- **影響ファイル**: <関連ファイル>
- **状態**: [TODO]
```

**Cの場合 → `COMPETITION.md` に追記**（テンプレートには含めない旨を明示）

---

### フェーズ3: テンプレートの進化方向の確認（思考層）

記録後、現在のTODO_TEMPLATE.mdの[TODO]項目を表示し、以下を問いかける:

「これまでに記録された改善の中で、**プロセス系（A）とインフラ系（B）**はいくつありますか？
もしコンペ固有（C）が多くなっていたら、テンプレートがこのコンペに特化しはじめているサインです。」

---

## TODO_TEMPLATE.md の状態タグ

- `[TODO]`: 未着手
- `[IN PROGRESS]`: 作業中
- `[DONE]`: 完了（mainブランチにマージ済み）
- `[DEFERRED]`: 将来の課題（今すぐは不要）
- `[REJECTED]`: 採用しないと判断（理由も記録する）

---

## 設計の意図

- **汎用/固有の峻別が最重要**: コンペを重ねるほどテンプレートがコンペ色に染まっていく。このスキルがその歯止めになる
- **発見のきっかけを残す**: 「何を改善するか」だけでなく「なぜ気づいたか」を残すことで、他のコンペでも同じ発見ができるかを判断できる
- **REJECTED の記録**: 「採用しないと判断した改善案」も残す。同じアイデアを何度も検討することを防ぐ
