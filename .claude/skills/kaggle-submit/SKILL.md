# /kaggle-submit スキル

## このスキルの役割

提出を「実験サイクルの完結点」として扱う。
何を変えたか → 提出 → OOF/LB比較 → 学びの言語化 → 次の仮説更新、
という流れを一連で行い、「提出したけど何も学ばなかった」を防ぐ。

---

## 実行手順

### フェーズ1: 提出前の文脈確認（思考層）

1. `experiments/log.csv` から今回の実験の `experiment_question`（目的）を表示する
2. 「今回の変更を1文で説明してください」とユーザーに問う
   - 目的に対して変更が整合しているか確認する
   - ズレがある場合は「実験目的は〈X〉でしたが、今回の変更〈Y〉はそれに対応していますか？」と確認する
3. 現在のOOF変化量を確認して提示する（前回比）

### フェーズ2: 提出実行（実行層）

1. `git status --short` 確認
   - working treeが汚れている場合: 「未コミットの変更があります。コミット後に提出してください。」と表示して停止

2. **提出枠の状況を確認して提示する**
   ```bash
   # 当日の提出回数を確認
   kaggle competitions submissions -c <COMPETITION> | grep $(date +%Y-%m-%d) | wc -l
   # コンペの締め切りを確認
   kaggle competitions list --search <COMPETITION>
   ```
   以下の形式で現状を提示する:
   ```
   ── 提出枠の状況 ──────────────────────
   本日使用済み: X / 5 回  （残り Y 回）
   コンペ締め切り: YYYY-MM-DD HH:MM (JST)
   残り日数: N 日
   推定残り総提出枠: Y + (N-1) × 5 = Z 回
   ──────────────────────────────────────
   未提出の候補ファイル（スコア順）:
     sub_171_lgb_cb_blend_0.91777_...csv
     sub_169_lgb_orig_proba_0.91760_...csv
     ...
   ```
   > **残り枠は使い切ることを基本方針とする。**
   > 未提出のまま締め切りを迎えた提出枠はゼロ価値。
   > ただし本日の枠が残り1回の場合は「最も有望な候補1本に絞る」判断を促す。

3. **⚠️ 実行前の最終確認（省略禁止）**

   以下の形式でユーザーに確認を取る。**ユーザーの「よろしいですか？」への明示的な返答を受けてから実行する:**

   ```
   `sub_NNN_model_0.XXXXX_YYYYMMDD_HHMM.csv` を提出します。
   本日 X/5 回目の提出になります。よろしいですか？
   ```

   - ユーザーが事前に「提出します」「試します」と言っていた場合でも、この確認は**省略しない**
   - 「提出します」= ユーザーの意思表示。AIの実行許可ではない
   - ユーザーの「OK」「はい」「どうぞ」を受けてから以下を実行:

   ```bash
   kaggle competitions submit -c <COMPETITION> -f data/output/submissions/<ファイル名> -m "<変更の説明>"
   ```

4. 数分後にLBスコアを確認する
   ```bash
   kaggle competitions submissions -c <COMPETITION> | head -5
   ```

5. **本日の残り枠がある場合**: 「あと Y 回提出できます。未提出の候補はありますか？」と確認する

### フェーズ3: OOF/LB 比較と振り返り（思考層）

スコアが判明したら以下を提示する:

```
今回の実験結果
  OOF変化: <前回比 ±X>
  LB変化:  <前回比 ±Y>
  現在ベストLB: Z
```

**OOF/LBの差分パターンに応じて問いかけを変える:**

| パターン | AIのコメントと問い |
|---|---|
| OOF↑ & LB↑ | 「OOFの改善がLBに反映されました。変更は汎化しています。なぜ汎化したと思いますか？」 |
| OOF↑ & LB→（横ばい） | 「OOFは改善しましたがLBに反映されていません。過学習またはCV設計のズレが考えられます。心当たりはありますか？」 |
| OOF↑ & LB↓ | 「OOFが改善してもLBが悪化しました。今回変えた〈X〉が汎化を損なっている可能性があります。なぜそうなったと思いますか？」 |
| OOF→ & LB↑ | 「OOFは変わらずLBが改善しました。OOFが捉えていなかった何かをLBが見ている可能性があります。」 |
| OOF↓ & LB↓ | 「両方悪化しました。〈撤退基準〉に達していますか？次のアクションを決めましょう。」 |

**学びの言語化:**
「この実験から何を学びましたか？次に活かせることを1〜2文で教えてください。」

アンサンブル実験で改善がなかった場合は、以下の棄却分析を追加で実施する:

「効かなかった理由はどれに当てはまりますか？」
- **同一予測（相関 ≥ 0.998）**: 「どうすれば予測の多様性を生み出せますか？」
- **誤差の方向が同じ**: 「どのサンプルで既存モデルと同じ間違いをしていますか？」
- **全モデルで改善なし**: 「これはFEに戻るシグナルですか？それとも問題の上限に近づいていますか？」

→ 棄却理由と次の方向性をセットで `learning` 列に記録する

→ ユーザーの回答を `experiments/log.csv` の `learning` 列に記録する

### フェーズ4: 次サイクルへの接続（思考層）

1. `log.csv` の該当行に `submit_score`・`lb_rank`・`learning` を更新する

2. **`learning` は必ず記入してから終了する。空欄のまま次の実験に進まない。**
   - 記入が難しい場合は以下の問いかけで引き出す:
     - 「OOFの変化（Δ〈X〉）はLBに反映されましたか？されなかった場合、なぜだと思いますか？」
     - 「この実験で『予想通り』だったこと・『予想外』だったことは何ですか？」
     - 「次の実験の方針はこの結果で変わりましたか？どう変わりましたか？」
   - 一言でもよい。「OOF↑LB→。CV設計のズレか外れ値の影響を次に確認する」でも記録として十分。

3. 学びを踏まえて問いかける:
   「この学びを踏まえると、次に検証すべき仮説は何でしょうか？」

4. **`SESSION.md` を更新する**（セッション引き継ぎのため）
   - `最後に完了したこと`: 実験ID・LBスコア・learning を記録
   - `次にやること`: ユーザーが答えた「次に検証すべき仮説」を記載
   - `現在の主要スコア`: 新しいベストスコアがあれば更新
   - `未解決の問い・ブロッカー`: OOF/LBギャップが説明できていない場合などを記録
   - `今サイクルで決めた重要な方針`: この実験の学びから方針変更があれば追記

5. 次の実験の方向性が決まれば `/new-experiment <次の実験名>` へ誘導する

---

## 注意事項

- このスキルは**ユーザーが明示的に呼び出した場合のみ**実行する
- `kaggle.json` が `~/.kaggle/` に存在することが前提
- フェーズ3の問いかけは省略しない。スコアだけ記録して終わることを防ぐのがこのスキルの存在意義
- **`learning` 列が空欄のまま次の実験に進むことを許可しない**
