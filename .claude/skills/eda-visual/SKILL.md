# /eda-visual スキル

## このスキルの役割

EDAを「データの全体像を自動生成する」作業から「自分が知りたいことをデータに問う」プロセスに変える。

生成されるダッシュボードより、**それを作る前の対話の方が価値がある**。
ユーザーが持ち込むドメイン知識と問いが、後続のFE仮説・モデル設計の質を決める。

---

## 実行手順

### フェーズ1: 問いの言語化（思考層）← 可視化より先に行う

データを読み込む前に、ユーザーに以下を問いかける。

---

**Q1: このデータが生まれた「文脈」を教えてください**

「何を目的として集められたデータか」「誰が・どんな行動をした記録か」を1〜2文で。

→ この文脈が「なぜターゲットが1になるか」の解釈基準になる。

**Q2: ターゲット変数について、何を「当然そうなる」と思っていますか？**

専門知識・業務経験・常識的直感、なんでも構わない。

→ これが「EDAで確認すべきこと」のリストになる。予想と結果の差分が最も価値ある発見になる。

**Q3: データに対して「気になること」「不安なこと」はありますか？**

（例: 「欠損値が多そう」「Train/Testで分布が違うかも」「特定のカテゴリが極端に少ないかも」）

→ これが「データ品質チェックの優先順位」になる。

**Q4（合成データコンペのみ）: 元データ（生成元）は入手可能ですか？**

合成データコンペ（Kaggle Playground Series等）では、元データとコンペデータの分布差が重要な情報を持つ。

→ 入手可能な場合: 「ターゲット率・数値分布・カテゴリ分布を元データとコンペデータで比較する」をフェーズ2に追加する
→ 比較のポイント: ①カテゴリ別ターゲット率の差、②数値変数の平均・分散の差（Cohen's d が大きい列が外部シグナルとして有効な候補）

---

ユーザーの回答を受けて、AIは以下を整理して確認する:

```
理解したいこと（Q2から）:
  1. <ユーザーが「当然そうなる」と思う変数 × ターゲットの関係>
  2. <ドメイン知識から重要と思われる数値変数とターゲットの関係>
  ...

確認すべきデータ品質（Q3から）:
  1. <欠損値が疑われる列の確認>
  2. <Train/Testで分布差が懸念される列の比較>
  ...

元データとの比較（Q4 — 合成データの場合）:
  1. カテゴリ別ターゲット率の差（元 vs コンペ）
  2. 数値変数ごとのCohen's dによる分布差スコア
  ...
```

「この理解で合っていますか？他に追加したい問いはありますか？」と確認する。

---

### フェーズ2: データ概要の把握（実行層）← 解釈はしない

対話を始める前に、全体像を素早く把握するための数値のみを収集する。
**ここで解釈・コメントをしない。数値を揃えるだけ。**

```python
# 必須（常に実施）
- ターゲット分布・クラスバランス（値と比率）
- 欠損値: 列ごとの欠損率一覧
- Train/Test サイズと列構成の確認

# 概要把握
- 数値変数の基本統計（min/mean/median/max/std）
- カテゴリ変数のユニーク数・最頻値
```

収集後、フェーズ1で決めた「問いの優先順位」に従い、**1変数ずつインタラクティブループへ入る。**

---

### フェーズ3: インタラクティブ可視化ループ（思考層 + 実行層）

> **原則: グラフを見せてからユーザーに先に語ってもらう。AIが先に解釈しない。**

フェーズ1の優先順位に従い、1変数（または1テーマ）ずつ以下のループを繰り返す。

```
─────────────────────────────────────────
STEP A【可視化生成】
  変数の種類に応じてグラフを生成・表示する:
  - 数値 × ターゲット: KDE/箱ひげ図の重ね合わせ
  - カテゴリ × ターゲット: ターゲット率の棒グラフ（件数も併記）
  - 数値 × 数値: 散布図（ターゲットで色分け）
  - Train vs Test: 分布の重ね合わせ

  ← 数値の羅列ではなくグラフ。グラフなしに次に進まない。

STEP B【ユーザー先行】
  グラフを提示した後、AIはすぐに解釈を述べない。
  まず問いかける:
  「このグラフを見て、最初に気になったことを教えてください」

  ← ユーザーが語るまで待つ。沈黙を埋めない。

STEP C【AIの深掘り】
  ユーザーの発言を受けて、以下のいずれかを実施する（複数可）:

  ① 裏付け確認
    「その解釈を補強する数値を確認してみましょう。〈クロス集計や追加グラフ〉」

  ② 多仮説の提示
    「このパターンの説明として〈仮説A〉〈仮説B〉〈仮説C〉が考えられますが、
     ドメイン的にはどれが最も説得力がありますか？」
    ← AIが1つの解釈に誘導しない。ユーザーのドメイン知識で選ばせる。

  ③ 逆仮説の提示
    「一方で〈反対の解釈〉という可能性はどうでしょうか？
     それを否定するにはどんなデータが見られるはずですか？」

  ④ 発見の接続
    「この観察は先ほどの〈前の発見〉と関係しているかもしれません。
     なぜなら〈接続の理由〉。2つを合わせるとどう解釈できますか？」

  ← ①〜④の中で「ユーザーの視野を広げるもの」を優先して選ぶ。
  ← 既にユーザーが言ったことを繰り返す確認作業はしない。

STEP D【欠損値テーマのみ: メカニズム分析】
  欠損のある列を扱う場合は以下を追加する:

  欠損の3分類を提示して判断を求める:
  | 種別 | 意味 |
  |---|---|
  | MCAR（完全ランダム欠損） | 欠損の有無が他の変数と無関係 |
  | MAR（条件付きランダム欠損） | 欠損の有無が他の観測変数と関係する |
  | MNAR（非ランダム欠損） | 欠損の有無が欠損値自体と関係する |

  「この列が欠損しているのは、どんなケースだと思いますか？」
  「欠損していること自体がターゲットと関係する可能性はありますか？」
  → MNAR と判断された場合: `is_missing_<列名>` フラグ特徴量の仮説をSTEP Eで登録する
  → 処理方針（補完方法・フラグ追加の有無）をフェーズ6の EDA_SUMMARY.md に記録する

STEP E【FE仮説の即時登録】← スキップ禁止
  「気になる発見」または「予想外の発見」が1件でも出たら即座に:
  「この発見を `/fe-hypothesis <タイトル>` で今すぐ記録しますか？」
  → 記録するまで次の変数に進まない。

─────────────────────────────────────────
```

**Train/Test 分布差で乖離が見つかった場合（必須チェック）:**
- 「この乖離はCV/LBスコアの乖離につながる可能性があります。CVの分割方法で対処できますか？それとも特徴量・前処理側での対応が必要ですか？」と問いかける

---

### フェーズ4: ループ完了後の全体統合（思考層）

全変数のループが終わったら、発見全体を俯瞰して問いかける:

「今回のEDAを通じて、最も『意外だった』発見はどれですか？」
「複数の発見を組み合わせると、どんなストーリーが見えてきますか？」

→ ユーザーの答えを `EDA_SUMMARY.md` の「予想外だった発見」に記録する。

---

### フェーズ5: FE仮説の登録確認（思考層）

フェーズ3のループ内で随時登録してきた仮説を確認する。

「ループ中に登録した仮説は〈N〉件でした。まだ言語化できていない発見や気になる点はありますか？」

→ 残りがあれば `/fe-hypothesis <タイトル>` で追加登録してから次へ進む。
→ ゼロ件の場合は「EDAで気になる発見がなかったか、または対話が不十分だった可能性があります。フェーズ3のループをもう1巡しますか？」と確認する。

---

### フェーズ6: グラフの生成と読み込み（実行層）

> **marimoは使用しない。** スクリプトで画像を生成し、Claudeが読んで対話する。
> これにより「Claude がグラフを見ながら話す」形式が実現できる。

フェーズ3のループ内では、各変数について以下を実行する:

```python
# 例: scripts/visualize.py を使うか、インラインで記述
import matplotlib
matplotlib.use("Agg")  # 非インタラクティブモードで描画
import matplotlib.pyplot as plt
import seaborn as sns

fig, ax = plt.subplots(figsize=(10, 4))
# ... 描画処理 ...
fig.savefig("data/output/plots/eda_{変数名}_{テーマ}.png", dpi=120, bbox_inches="tight")
plt.close(fig)
```

保存後、Claudeは `Read` ツールで画像を読み込み、グラフを見ながらSTEP Bの問いかけを行う。

**命名規約**: `data/output/plots/eda_{変数名}_{テーマ}.png`
- 例: `eda_age_target_dist.png`、`eda_category_target_rate.png`、`eda_train_test_col1.png`

**ループ終了後（フェーズ4〜5が完了した後）、総括画像セットを保存する:**

```
data/output/plots/
  eda_target_distribution.png     ← ターゲット分布
  eda_missing_values.png          ← 欠損値ヒートマップ
  eda_train_test_compare.png      ← Train/Test 分布比較
  eda_{変数名}_*.png              ← 各変数の個別グラフ
```


#### 7-B: `FEATURE_REPORT.md` の「生変数サマリー」を記入

フェーズ3のループで確認した各変数を、`FEATURE_REPORT.md` の「生変数サマリー」テーブルに追記する:

| 記入項目 | 情報源 |
|---|---|
| 特徴量名・型 | データ概要（フェーズ2） |
| 欠損率 | フェーズ2の欠損値確認 |
| ターゲット相関 | STEP A・Bの可視化から判断 |
| Train/Test分布差 | STEP A の train_test グラフから |
| 欠損メカニズム | STEP D の MCAR/MAR/MNAR 分析 |
| 処理方針 | STEP D の対話で決定した方針 |

→ **FEATURE_REPORT.md の「生変数サマリー」が埋まることで、FE段階でどの変数が未探索かが一目で分かる。**

## 設計の意図

- **Claudeが画像を読む**: `matplotlib` で `Agg` バックエンドに保存 → Claudeが `Read` で読み込む。marimoのようにユーザーが画面を見て口頭で説明する必要がなく、Claudeと画像を「共有」した状態で対話できる
- **グラフが先、数値は補助**: 数値の羅列では「パターン」は見えない。グラフを見せてから対話を始める
- **ユーザーが先に語る**: AIが先に解釈を述べると、ユーザーはそれに引きずられる。グラフを見せたら沈黙して待つ
- **多仮説提示でドメイン知識を引き出す**: 1つの解釈を提示するより「A・B・Cどれが説得力ありますか？」の方がユーザーの知識が引き出される
- **発見を接続する**: 個別の観察を「前の発見と合わせると」という形で結びつけることで、ユーザーの視野が広がる
- **1変数ずつのループ**: 全データを一気に見せると情報過多で「なんとなく了解」になる。1変数ずつ完結させてから次へ
- **欠損値は補完前にメカニズムを議論する**: MNAR な欠損を中央値で補完するだけでは情報が失われる。欠損フラグが有力な特徴量になることがある
- **Train/Test 比較は必須**: 分布差を EDA 段階で把握しておかないと、CV/LB 乖離の原因特定が遅れる
- **FE仮説への即時接続**: 「発見したが仮説を立てずに直接実装」を防ぐ。記録するまで次に進まないことで強制力を持たせる
