# /fe-hypothesis スキル

## このスキルの役割

特徴量エンジニアリングを「試して見る」ではなく「因果を考えてから試す」プロセスに変える。
採用・棄却どちらの結果も、次の仮説をより賢くするための情報として構造化して残す。

**3つのモード:**
- `/fe-hypothesis <仮説タイトル>` — 新規仮説の立案・記録
- `/fe-hypothesis update H-NNN` — 検証結果の記録と学びの構造化
- `/fe-hypothesis list` — 仮説一覧の表示と次の候補レコメンド

---

## モード1: 新規仮説の立案

### フェーズ0: 既存知識との照合（思考層）

`FE_HYPOTHESES.md` の「確認済みパターン」セクションを読み込み、以下を確認する:

1. **類似仮説が既にないか** — 同じ特徴量・同じメカニズムの重複を防ぐ
2. **確認済み棄却パターンに当てはまらないか** — 「n不足」「既存FEと冗長」「モデル依存逆転」等
3. もし当てはまる場合は「この仮説は〈パターンX〉に似ています。それでも異なる理由はありますか？」と問いかける

### フェーズ1: 因果メカニズムの掘り下げ（思考層）

以下を順番に問いかける。ユーザーが答えるまで次に進まない。

---

**Q0: この特徴量が使う変数は、ドメイン上「何を意味する」量ですか？（ドメイン実体の確認）**

AI が因果を推測する前に、userのドメイン知識から変数の実体を引き出す。
「共創の原則」（CLAUDE.md）— AIはドメイン知識の代わりにならない。因果はドメインの実体から立ち上げる。

- 「この列は物理的/業務的に何を表しますか？単位・スケール・典型値は？」
- 「その量が大きい/小さいとき、対象は現実世界でどういう状態ですか？」
- 「ドメイン専門家ならこの変数をどう解釈しますか？」

> **なぜ Q0 が重要か**: 変数の実体を掴むと因果連鎖（Q1）が「相関がありそう」から
> 「この物理量がこう振る舞うからターゲットが決まる」という鋭い仮説に変わる。
> 過去事例では、ドメイン的に自明な変数（例: 天体の赤方偏移＝距離/種別の指標）の実体を
> 掴むことで、単純な統計特徴量では届かない外部シグナル系の FE に到達できた。
> user がドメインに不案内な場合は「この変数の意味を一緒に調べましょう」と提案し、
> データ辞書・コンペ説明・元データソースを確認する。

**Q1: この特徴量がターゲットと「なぜ」関係するか、因果の連鎖を説明してください**

「Aだから→Bになる→だからターゲットCが起きる」の形で表現できると理想的。
「相関がありそう」「効きそう」は因果の説明ではない。

（例: 「〈属性A〉の顧客は〈行動B〉を取りやすい→その結果〈状態C〉になる→ターゲットと強く関係する」）

**Q2: この仮説が**外れる**とすれば、どんな理由が考えられますか？（失敗モード）**

事前に失敗モードを考えることで、棄却後の学びが深まる。
- 「既存特徴量がこの情報を既に学習済みかもしれない」
- 「n が少なすぎて統計的な信号にならないかもしれない」
- 「因果の方向が逆かもしれない（解約が先で、この状態が後）」

**Q3: この特徴量はモデルによって効果が変わると思いますか？**

LGB（決定木系）とCB（Ordered Boosting）では特徴量の使い方が異なる場合がある。
- 「LGBには効くがCBには効かない可能性がある。なぜなら…」
- 「両モデルで同様に効くと思う。なぜなら…」

---

ユーザーの回答を受けて、AIは以下を確認する:
- Q1が「〜と相関する」になっていたら「その相関はなぜ起きるか、もう一段掘り下げられますか？」と促す
- Q2が「特になし」だったら「既存の〈類似特徴量〉と情報が重複する可能性はどうでしょう？」と具体的に提示する

**知識移転の確認（フェーズ2 に進む前に必須・スキップ禁止）:**

仮説を記録する前に、AI は仮説のメカニズムを**平易な言葉に翻訳して**ユーザーに説明し、理解を確認する:

- 専門用語（ドメイン固有の量・統計手法名）が含まれる場合は、毎回言い換えを添える
  （例: 「percentile特徴量 = この観測値が、参照データの各グループの中で何%目に位置するか」）
- 「この説明で、なぜこの特徴量が効きそうか腑に落ちますか？」と問い、
  ユーザーが**自分の言葉で実験結果を評価できる状態**になってから実装に進む
- ユーザーから異論・違和感が出たら、それは仮説を磨く材料。記録してから進む

> **理由**: 「なぜ効くか」が FE_HYPOTHESES.md に記録されていても、対話で共有されなければ
> ユーザーは実験結果を評価に参加できない。**記録 ≠ 知識移転**。
> AI がドメイン知識を持たないユーザーに代わって判断を進めてしまう構造（共創原則違反）をここで断つ。

### フェーズ2: FE_HYPOTHESES.md への記録 + SESSION.md 更新（実行層）

以下の拡張フォーマットで追記する:

```markdown
## H-NNN: <仮説タイトル>
- **ステータス**: 🔵 未検証
- **ドメイン実体**: <Q0: 変数がドメイン上何を意味するか。単位・スケール・現実世界の状態>
- **仮説**: <ユーザーの説明をもとにAIが整理>
- **メカニズム**: <Q1の因果連鎖を1-2文で（ドメイン実体から立ち上げる）>
- **特徴量案**: `feature_name = ...`
- **失敗モード**: <Q2の回答>
- **モデル依存性**: <Q3の回答（LGB/CB/両方）>
- **可視化確認**: — （フェーズ3で記録）
- **検証方法**: base CV + ΔOOF（scripts/feature_study.py）
- **結果**: —
- **棄却理由**: —
- **学び**: —
```

FE_HYPOTHESES.md への記録後、`SESSION.md` を更新する:
- `現在のステージ`: Stage 4 — 段階的FE（仮説 H-NNN を登録）
- `次にやること`: 「H-NNN を実装 → scripts/visualize.py で可視化確認 → scripts/feature_study.py でΔOOF計測」

**フェーズ2 完了後に必ず以下を出力する（省略禁止）:**

```
H-NNN「<仮説タイトル>」を FE_HYPOTHESES.md に記録しました。

次のステップ:
  1. 特徴量を実装する（`experiments/runs/expNNN_...` に実装コードを書く）
  2. **モデルに投入する前に** フェーズ3（可視化確認）を実施する
     → `scripts/visualize.py` または `scripts/feature_report.py` で分布を確認
  3. 可視化で問題なければ `scripts/feature_study.py` でΔOOF計測

フェーズ3（可視化確認）を今すぐ実施しますか？
（特徴量の実装が完了してから呼んでください）
```

この出力が表示されない限り、ユーザーはフェーズ3の存在に気づかない。
**ユーザーが「省略」「後で」と言った場合のみスキップ可。AIが自分の判断でスキップすることは禁止。**

---

### フェーズ3: 実装後・投入前の可視化確認（実行層 + 思考層）← スキップ禁止

**特徴量を実装したら、モデルに投入する前に必ずこのフェーズを実施する。**
実装バグや想定外の分布をここで発見することで、無駄なモデル学習コストを防ぐ。

```
STEP A【分布の確認】
  生成した特徴量の分布を可視化する:
  - 数値特徴量: ヒストグラム・箱ひげ図（外れ値・スケールを確認）
  - カテゴリ特徴量: 値ごとの件数（極端に少ないカテゴリがないか）
  - 欠損値の数を確認（想定外の欠損が生じていないか）

STEP B【ターゲット別分布の確認】
  特徴量 × ターゲットの関係を可視化する:
  - 数値: ターゲット別KDE・箱ひげ図の重ね合わせ
  - カテゴリ: ターゲット率の棒グラフ
  → 「仮説で想定したパターンと一致していますか？」とユーザーに問いかける

STEP C【ユーザーとの対話】
  可視化を見せてから問いかける（eda-visual と同じプロトコル）:
  「この特徴量の分布、想定通りに見えますか？」
  「仮説で予想していたターゲットとの関係が見えていますか？」

  → 想定と異なる場合: 実装を見直す（モデルに投入しない）
  → 想定通りの場合: STEP D へ

STEP D【冗長性の事前確認】
  既存の採用済み特徴量との相関を確認する:
  - 相関係数が 0.95 以上の既存特徴量がある場合: 「冗長になる可能性があります。
    それでも追加する理由はありますか？」と問いかける
  → 冗長と判断した場合: ここで棄却してモデル投入をスキップ（コスト節約）

STEP E【確認結果の記録】
  FE_HYPOTHESES.md の「可視化確認」フィールドに記録する:
  「分布: <状態>、ターゲット別パターン: <仮説と一致/不一致>、冗長性: <なし/要確認>」
```

可視化確認が完了したら `scripts/feature_study.py` でΔOOFを計測し、結果を `/fe-hypothesis update H-NNN` で記録する。

## モード2: 検証結果の記録（`/fe-hypothesis update H-NNN`）

### フェーズ1: 結果の確認

該当仮説のエントリを表示し、「事前の予測（成功・失敗モード）と結果を照合」する。

### フェーズ2: 採用の場合（思考層）

1. 結果・考察をエントリに記録してステータスを ✅ に更新
2. **`FEATURE_REPORT.md` の「エンジニアリング済み特徴量」テーブルに追記する:**
   ```
   | <特徴量名> | H-NNN | <メカニズム1行> | <ΔOOF値> | ✅ 採用 | <日付> |
   ```
3. **確認済みパターンへの追加**を提案する:
   - 「このメカニズム（〈X〉）は他の未検証仮説にも応用できそうですか？」
4. モデル依存性の検証を促す:
   - 「LGBで効きましたが、CBでも同様ですか？逆の可能性はありますか？」
5. `FEATURE_REPORT.md` の「現在の特徴量セット」セクションの採用件数・OOFを更新する

### フェーズ3: 棄却の場合（思考層）← ここが最も重要

結果と事前の失敗モードを照合し、棄却理由を以下の分類で記録する:

| 棄却パターン | 意味 | 例 |
|---|---|---|
| **冗長** | 既存FEが情報を学習済み | 「H-006: high_risk_tripleと重複」|
| **因果誤り** | メカニズムの前提が間違い | 「H-011: 解約は更新前ではなく更新後に集中」 |
| **n不足** | セグメントが小さすぎる | 「H-009: n=120で統計的信号にならない」 |
| **モデル依存逆転** | LGBで負、CBで正（または逆） | 「H-007: LGB -0.00007, CB +0.00003」 |
| **効果量不足** | 因果は正しいが改善幅が小さい | 「H-002: delta≈0で誤差範囲」 |

**棄却記録への必須追記（CLAUDE.md Stage 4 ルール）:**

棄却エントリには必ず「棄却したアーキテクチャ名」を明記する:
```
- 棄却: <アーキテクチャ名>（例: LGB）で ΔOOF=<値>（閾値未満）
- 未評価: <他のアーキテクチャ名>（例: RealMLP, CatBoost）
- 再試行条件: 主軸アーキテクチャが変わった場合は再評価する
```

> **理由**: あるアーキテクチャで棄却した FE が、別アーキテクチャでは有効なケースがある。
> 「LGB 棄却 = 全アーキテクチャで棄却」という誤判断を防ぐ。

棄却パターンを記録した後、**必ず以下を問いかける**:

**「この棄却は他の未検証仮説に何か示唆しますか？」**
- 冗長で棄却 → 「同じ特徴量を使う他の仮説も再考が必要では？」
- モデル依存逆転 → 「LGB/CB最適特徴量セットを分けて管理すべきかもしれません」
- n不足 → 「セグメント定義をより広く取り直せますか？」

この「前向きな学び」も棄却エントリに記録する。

また **`FEATURE_REPORT.md` の「エンジニアリング済み特徴量」テーブルに棄却行を追記する:**
```
| <特徴量名> | H-NNN | <メカニズム1行> | <ΔOOF値> | ❌ 棄却（<理由略記>） | <日付> |
```

### フェーズ4: 次のアクションへの接続（採用・棄却共通）

update 完了後に必ず以下を提示する:

**採用の場合:**
```
✅ H-NNN を採用記録しました。

次のアクション:
  A) さらに仮説を検証する → /fe-hypothesis list（次の候補を確認）
  B) 特徴量セットが十分なら本格HP最適化へ → scripts/optimize_hp.py --n-trials 150 --tag full
  C) 新しい実験として実行記録する → /new-experiment <実験名>
```

**棄却の場合:**
```
❌ H-NNN を棄却記録しました。

次のアクション:
  A) 次の仮説を確認・選択する → /fe-hypothesis list
  B) 棄却の示唆から新しい仮説を立てる → /fe-hypothesis <新仮説タイトル>
  C) FEが十分に探索済みなら次のステージへ → /new-experiment <実験名>
```

SESSION.md の「次にやること」を更新する（採用なら特徴量数・OOF を、棄却なら次の候補仮説 ID を記録）。

## モード3: 仮説一覧の表示（`/fe-hypothesis list`）

1. ステータス別に整理して表示する
2. **確認済みパターンのサマリーを表示する:**

```
【確認済みメカニズム】
  - <採用された仮説から抽出した、再利用可能なメカニズム（H-NNN で確認）>
  - ...

【確認済み棄却パターン】
  - <棄却理由の分類と示唆（例: n<200のセグメントは統計的信号にならない）>
  - ...

【次の推奨仮説の方向性】
  - 未探索の角度: <EDAから示唆される未検証の軸>
  - 既存棄却の再検討候補: <条件を変えれば復活できる仮説>
```

3. 未検証仮説のうち次に検証すべきものを理由付きでレコメンドする

## FE_HYPOTHESES.md の「確認済みパターン」セクション

このスキルが管理する FE_HYPOTHESES.md には、冒頭に以下のセクションを維持する:

```markdown
## 確認済みパターン

### 有効なメカニズム
<!-- 採用された仮説から抽出した、再利用可能な発見 -->

### 棄却パターン（繰り返しを防ぐ）
<!-- 棄却理由の分類と、その示唆 -->

### モデル別の傾向
<!-- LGB/CBで異なる挙動を示す特徴量の記録 -->

### 外部データ活用（合成データコンペの場合）
<!-- 元データから取得した統計量を特徴量化した結果を記録 -->
<!-- パターン例: カテゴリ別ターゲット率、数値分布との距離 (z-score/percentile/Euclid) -->
```

このセクションは仮説の追加・更新のたびに最新状態に保つ。

## 設計の意図

- **失敗モードの事前言語化**: 棄却後に「やっぱりこうなったか」と気づくより、事前に書いておく方が学びが深い
- **棄却パターンの分類**: 「効かなかった」で終わると同じ失敗を繰り返す。「なぜ効かなかったか」の類型が次の仮説の質を上げる
- **モデル依存性の明示**: LGB/CBで異なる特徴量最適セットが存在することをこのスキルで系統的に追跡する
- **確認済みパターンの維持**: 15件以上の仮説が蓄積されると「以前同じことを試したか」が分からなくなる。パターンの抽象化が探索効率を上げる
- **実装後・投入前の可視化確認（フェーズ3）**: 実装バグ・想定外の分布・冗長性をモデル投入前に発見する。「ΔOOFが低かった原因が特徴量の問題かモデルの問題か」の切り分けにも使える。ユーザーのドメイン知識を「特徴量を見ながら」引き出す最後の機会
